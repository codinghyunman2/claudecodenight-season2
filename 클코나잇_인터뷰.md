> 발표 포지션: 비개발자 · 내 업무 하나를 통째로 에이전트에게 넘겨본 이야기

## 1. 자기소개 & 배경

**Q1. 간단한 자기소개 부탁드려요. 어떤 팀에서 어떤 일을 하시나요?**

> 스파르타 게임즈에서 BizOps Lead로 마케팅·사업·CS·피플 4개 파트를 담당한다는 것, 비개발자라는 점을 자연스럽게 연결해서 답변.

---

**Q2. Claude Code를 시작하기 전에는 어떤 도구를 쓰셨나요? 왜 Claude Code로 전환하셨어요?**

> 대화형 → Genspark → Manus → n8n → Claude Code 순으로 변화한 이유와 차이점 설명.  
> 단순 사용 편의가 아니라, "코드로 로직을 직접 설계할 수 있게 됐다"는 방향으로 답변.

---

## 2. 발표 핵심 주제 검증

**Q3. "내 업무 하나를 통째로 넘겼다"고 하셨는데, 가장 대표적인 사례를 하나만 고른다면요?**

> 데일리 캠페인 퍼포먼스 리뷰(#3)가 가장 적합 — 판단 기준 수립 → 자동화 → 팀 사용까지 흐름이 완결됨. "매일 오전 60분 소요되던 작업이 0분이 됐고, 지금은 결과를 보고 판단만 한다"는 식으로 구체화.

---

**Q4. 13개 자동화 중 9개가 팀에서 매일 돌아간다고 하셨어요. 나머지 4개는 왜 팀에서 안 쓰이나요?**

> "나만 쓰는 도구가 됐다"는 실패담과 연결. 내 판단 기준이 과하게 들어가 타인이 맥락 없이 쓰기 어려웠던 케이스 솔직하게 설명. Recruitment 스킬이나 MktPerformanceAgent 등 실험 단계 도구 언급.

---

**Q5. "AI 활용의 병목은 기술이 아니라 문제 정의 능력이다"라는 인사이트는 어떤 경험에서 나왔나요?**

> "ROAS 30% 하락 시 주의"라는 판단 기준이 말로 정리되기 전까지는 코드가 아무것도 해줄 수 없었다는 구체적 에피소드. 
> 도메인 지식 → 판단 기준 → 규칙 문서화 → 자동화 흐름으로 답변.

---

## 3. 실패담 구체성

**Q6. 파일명 규칙 없이 자동화를 먼저 만들었다가 소재 오분류가 발생했다고 하셨는데, 실제로 어떤 피해가 있었나요?**

> 광고 집행 데이터 엉킴 → 파싱 규칙 전면 재설계 → 며칠치 리포트 수동 복구 같은 구체적 후속 비용 언급. 
> "자동화 전에 데이터 구조 설계가 먼저다"는 교훈으로 마무리.

---

**Q7. YouTube API 한도 문제로 업로드가 멈췄던 상황, 좀 더 자세히 말씀해 주실 수 있어요? 그때 어떻게 해결하셨나요?**

> YouTube Data API v3 일일 할당량(quota) 초과 → 크리에이티브 업로드 중단 → Google 측 한도 증량 요청 프로세스. 
> "사전에 제약 조건을 인지하고 설계에 반영해야 한다"는 교훈, 이후 업로드 수 제한 로직 추가 여부까지 답변 가능하면 플러스.

---

**Q8. 실패를 겪으면서 Claude Code 사용 방식 자체가 바뀐 게 있나요?**

> API 문서를 먼저 읽히는 습관(컨텍스트 주입), 데이터 구조 설계를 먼저 문서화하는 루틴, 팀 공통 기준으로 먼저 정리하고 개인 판단 기준은 나중에 넣는 방식 등으로 구체화.

---

## 4. 팀 전파 & 정착

**Q9. 팀원들이 처음에 자동화 결과를 어떻게 받아들였나요? 저항이나 무관심은 없었나요?**

> 슬랙으로 결과를 내보낸 것이 결정적 — "별도 도구 학습 없이 기존 루틴에 끼워 넣었다"는 접근법 설명. 
> 초기 반응과 현재 9개 운영까지의 과정에서 구체적 에피소드 있으면 언급.

---

**Q10. 비개발자가 Claude Code로 팀 자동화를 만들면 "유지보수"는 어떻게 하세요? 뭔가 깨지면 어떻게 고치나요?**

> 실제 유지보수 경험 — Slack 알림이 안 올 때, API 스펙이 바뀔 때 등. Claude Code에 에러 메시지 붙여넣기 → 수정 요청 패턴, 또는 에러 처리를 설계 단계에서 명확히 해뒀기 때문에 대부분 자가 복구된다는 식으로 답변.

## 5. 발표 형식 & 준비도

**Q11. 15분 안에 어떤 순서로 이야기를 구성할 생각이신가요?**

> 슬라이드 구성 예시: 배경(2분) → 대표 자동화 사례 1개 딥다이브(4분) → 실패담 2개(4분) → 인사이트 + 청중 takeaway(3분) + 질의(2분). 
> 현재 portfolio.html 슬라이드 12장 구조 기반으로 답변 가능.

---

**Q12. 청중 중에 "나는 비개발자라 못 할 것 같다"고 생각하는 분들에게 가장 전달하고 싶은 한 마디는요?**

> "코딩 능력이 없어도 됩니다. 논리적으로 문제를 정의할 수 있으면 됩니다."  
> — 직무 전문성이 있는 사람이 AI를 가장 잘 씁니다. 이 메시지로 마무리.

## 보너스 — 날카로운 압박 질문

**Q13. 272시간 절감이라는 수치, 어떻게 산출하셨나요? 실제로 그 시간을 다른 업무에 쓰고 있나요?**

**Q14. 13개 자동화 중 지금 당장 없애도 가장 아깝지 않은 게 어떤 건가요? 그 이유는요?**
