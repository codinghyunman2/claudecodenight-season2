## 2. 게임업계 맥락 (1분) — 신규

### 슬라이드 03 — "4배 늘어난 업무는 게임업계라서 더 빨리 왔습니다"

| Pressure | 게임업계 특유의 상황 | 어떤 자동화로 이어졌나 |
|----------|---------------------|------------------------|
| **01 · 빠른 출시 사이클** | 소프트런칭 / 글로벌론칭 / LiveOps가 동시 진행. 게임마다 일정 · KPI가 다 다름 | 제품 캘린더 대시보드, 피처링 요청서 자동화 |
| **02 · 유저 리뷰 민감도** | 별점 1~2점이 ROAS · CPI에 즉시 영향. 부정 리뷰 급증 감지가 늦으면 광고비가 손실로 전환 | 리뷰 답변 초안 자동 생성, 리뷰 트렌드 슬랙 알림 |
| **03 · 소재 회전율 · 매트릭스** | 4게임 × 4국가 × 2타입 = 32개 폴더를 매주 점검. 광고 소재 수명이 짧아 회전이 잦음 | 소재 재고 관리 자동화, 주간 CVR 트래커 |

### 발화 포인트

> "방금 '4배 늘어난 업무'라고 말씀드렸는데, 이건 게임업계에서 일하는 분들이라면 더 공감하실 겁니다.
>
> 출시 사이클이 짧고 동시다발적입니다. 소프트런칭, 글로벌론칭, LiveOps가 동시에 진행되니까요. 그리고 유저 리뷰 한 두 개가 광고비를 즉시 손실로 바꿉니다. 게다가 광고 소재는 수명이 짧아서 매주 4게임 × 4국가 × 2타입, 총 32개 폴더를 점검해야 합니다.
>
> 이 세 가지 압박이 4배 늘어난 업무량과 합쳐졌을 때, 자동화는 **'효율'이 아니라 '생존'**의 문제가 됐습니다. 제가 만든 도구들의 우선순위가 왜 이렇게 짜였는지 — 이 맥락이 답입니다."

## 3. 도구 변천사 (1.5분)

### 발화 포인트

- 처음부터 클로드 코드로 시작한 게 아님

- **요즘IT 분들께 드리는 솔직한 말씀**: "지금 당장 실무 적용이 목표라면 도구에 얽매이지 마세요"
  - Google Apps Script만으로도 메일 정리 / 시트 자동화 충분
  - Genspark, Manus 같은 노코드 빌더로도 빠르게 시작 가능
  - n8n으로 트리거·웹훅 연결까지 무리 없음
  - **클로드 코드는 "직접 로직을 설계하고 싶을 때" 비로소 필요한 단계**

### 강조

> "도구를 옮겨갈 때마다 새로운 도구를 배운 게 아니라, **이전 도구의 한계를 알게 됐다**는 게 핵심입니다. 한계를 모르면 다음 도구를 잘 쓸 수도 없습니다."

## 5. ROI 큐레이션 (1.5분) — 신규

### 슬라이드 09 — "임팩트가 컸던 3개, 의외로 어려웠던 3개"

**ROI Top 3 (월간 절감 시간 기준)**
1. **데일리 캠페인 리뷰** — Airbridge → 분석 → Slack 자동 발송 / **월 45시간 절감**
2. **Rocket Brief** — RSS 15채널 → Claude 번역 → 9시 슬랙 발송 / **월 15시간 절감**
3. **피처링 요청서** — `/pptx` 스킬로 14장 자동 생성 / **건당 3h → 30m**

**예상 외 실패 3 — "기능은 OK, 운영에서 부서지더라"**
1. **파일명 규칙 부재** — 소재 오분류 발생 → 이후 파싱 규칙 전체 재설계
2. **API rate limit 미숙지** — YouTube Data API 한도 도달 → 업로드 중단 → 한도 증량 요청
3. **"나만 쓰는" 도구** — 내 판단 기준 과다 주입 → 맥락 없이 타인이 쓰기 어려워 팀 미사용

### 발화 포인트

> "26개를 만들어 보니까 ROI 격차가 굉장히 컸습니다.
>
> 임팩트가 컸던 Top 3은 모두 **매일·매주 반복되는 데이터 정리**였습니다. 데일리 캠페인 리뷰가 월 45시간, Rocket Brief가 월 15시간, 피처링 요청서가 건당 3시간을 30분으로. 공통점은 '판단 기준을 한 번 문장으로 정리하면, 그다음부터는 AI가 매일 같은 품질로 처리한다'는 거였습니다.
>
> 반면 의외의 실패도 있었습니다. 기능은 멀쩡한데 운영에서 부서지는 패턴입니다.
> 첫째, 파일명 규칙을 안 잡고 자동화부터 만들었더니 소재가 오분류됐고, 파싱 규칙을 전체 재설계해야 했습니다.
> 둘째, Meta API rate limit을 모르고 만들었다가 업로드가 멈췄고, 사후에 한도 증량을 요청해야 했습니다.
> 셋째, 제 판단 기준이 너무 많이 들어간 도구는 결국 저만 썼습니다.
>
> 그래서 저는 ROI를 **'시간 절감 + 팀이 매일 쓰느냐'** 두 축으로 봅니다. 시간만 줄였는데 저만 쓰면 사실은 ROI가 아니거든요."

## 6. 핵심 인사이트 1 — 기획이 8할이다 (3분)

### 슬라이드 6 — "PM으로서 다시 느낀 기획의 중요성"

- 26개를 만들면서 가장 시간을 많이 쓴 건 코드가 아니라 **기획**
- 기획이 흔들리면 코드는 무조건 갈아엎어야 함 → 리소스 낭비

### 슬라이드 7 — 기획 3단 세팅

**1단계 — 문제정의**
- *"왜 자동화하는가? 진짜 병목이 어디인가?"*
- 예시: "리뷰 답변 자동화"가 아니라 **"별점별 톤을 매번 머리에서 재구성하는 비용 제거"**가 진짜 문제였음

**2단계 — PRD 작성**
- 노션/구글닥에 1~2장 분량으로 정리
- 입력 / 출력 / 판단 기준 / 실패 시 처리 — 4가지만 명확히

**3단계 — CLAUDE.md 기본 세팅**
- 프로젝트 루트에 CLAUDE.md를 둔다 → 컨텍스트가 매번 유지됨
- 도메인 용어, 판단 기준, 폴더 구조, 절대 하지 말 것 — 이 4가지를 기록
- *"내가 매일 클로드에게 똑같은 설명을 반복하고 있다면, 그건 CLAUDE.md에 들어가야 할 내용"*

### 슬라이드 8 — 기획 변동 리스크 최소화

- 중간에 스펙이 바뀌는 건 **정상이고 불가피**함
- 단, 코어 로직(판단 기준)이 흔들리지 않도록 PRD를 변경 이력으로 관리
- 작은 단위로 쪼개서 만들 것 — MVP로 먼저 굴리고, 검증 후 확장

### 강조

> "AI는 문제해결 도구입니다. 기존 PM 프레임워크는 여전히 유효합니다. **달라진 건 실행 속도뿐.** 기준이 없으면 AI는 방향 없이 빠를 뿐입니다."

## 7. 핵심 인사이트 2 — 용이함 / 확장성 / 보안 (2.5분)

### 슬라이드 15~16 — 비개발자가 챙겨야 할 3축

**용이함**
- 비개발자이기에 **접근성이 최우선**
- SSOT(Single Source of Truth)가 될 수 있는 도구를 선택
- 팀원이 접근해야 하는지 먼저 판단 → 꼭 멋드러진 DB가 필요한 건 아님
- **스프레드시트로 꽤 많이 처리 가능** — MCP 연동되어 있으면 시트만 바꿔도 프로덕트 반영
- MVP로 먼저 만들자

**확장성**
- MVP 이후를 염두에 둬야 함 — 갈아엎지 않을 수준으로
- 그 관점에서도 **스프레드시트가 의외로 유효**
  - 프로젝트가 늘어날 때
  - 트래킹 지표가 추가될 때
  - 스펙 변경이 잦은 항목은 처음부터 시트 기반으로 빼두기

**보안** (사내 프로젝트는 대부분 대외비)
- 깃허브 Public 필요 없음. **Private 레포가 디폴트**
- 배포 방식 결정 기준:
  - **Daily 지표 체크** → GitHub Actions
  - **Monthly 점검** → 로컬 실행
  - **상시 팀원 공유 필요** → Vercel 배포 (커스텀 도메인은 일부러 안 붙임)
- 코드 단에서는 항상 `security-review` 활용 → **우선순위 중 이상은 반드시 반영**
- 최소 기준: **브라우저 콘솔에 이슈 안 나오게**
- 최소한의 OAuth: 공유 비밀번호 + Middleware 쿠키 형태로도 충분
- 잘 몰라도 클로드 코드에 상황 공유하고 개선점 요청 → 리스트 보고 의사결정만 하면 됨

### 슬라이드 17 — 보안 체크 프롬프트 예시

```text
아래 코드를 Vercel 배포 전 보안 리뷰해줘.
[코드 붙여넣기]

스택: Next.js App Router + Vercel + [DB/Auth 명시]

각 항목을 현재 코드 기준으로 진단:
1. CORS  2. CSRF  3. XSS+CSP  4. SSRF
5. AuthN/AuthZ  6. 최소권한  7. Input Validation + SQLi
8. Rate Limit  9. 쿠키·세션  10. Secret 관리
11. HTTPS/HSTS·보안헤더  12. Audit Log
13. 에러 노출  14. 의존성 취약점

출력: | 항목 | 상태 | 위험도 | 수정 방향 |
상태: ✅ 구현 / ⚠️ 부분 / ❌ 없음 / ➖ 해당없음
위험도: High / Med / Low
```

> 이 프롬프트 하나만 챙겨가셔도 오늘 발표의 본전은 뽑으십니다.

## 8. 도구는 거들 뿐 — 외부 플러그인 & 자체 스킬 (1.5분)

### 슬라이드 18~19 — "쓸 만한 건 일단 스크랩, 적용은 천천히"

**얼리어답터 팔로우 → 스크랩 → 선별 적용**
- 스레드에서 활동하는 얼리어답터 분들 팔로우
- 추천 받은 것 중 **필요해 보이는 건 일단 스크랩**
- 추후 하나씩 적용해보며 판단 — 계속 쓸지, 버릴지
- 그 경험들이 쌓여야 **나만의 것**을 구축할 수 있음
- 본인의 시작은 **구글 메일함 정리** (2만 개 쌓여 있어서)
